# Code Contract

This repository follows this contract.

All contributors and assistants must obey this contract.
If a request conflicts with this contract, the contract wins.

The goal is to ship correct, readable, maintainable systems with strict quality gates.

No emojis.

---

# 1. Workflow and CI

Code must pass local checks before any push.
Code must pass CI.
No partial compliance.
No “fix later.”

Local checks must enforce:

* Format
* Lint
* Type checking
* Tests
* ASCII rule
* Exception rule
* Placeholder rule
* Documentation placement rule
* Archive rule

Failure in any gate blocks completion.

---

# 2. Dependency and Schema Control

No new dependency without explicit approval.

No database schema change without a written migration plan.

A migration plan must include:

* Forward migration steps
* Backward compatibility strategy
* Rollback plan
* Affected modules

---

# 3. Project Structure

Each module owns one domain.

Separation of concerns is mandatory:

* IO lives in IO modules
* Parsing lives in parsing modules
* Business logic lives in service modules
* Data models live in model modules

Forbidden:

* God modules
* Catch-all util.py
* Circular imports
* Deep nesting

Names must encode intent and units where relevant.

Examples:

`load_prices_usd_daily`
`compute_drawdown_pct`
`forecast_demand_mwh_hourly`

---

# 4. Python Standards

Code must be idiomatic and Pythonic.

Prefer:

* Dictionary dispatch over long if/elif chains
* Vectorized logic over condition trees
* Small pure functions
* Early returns
* pathlib over os.path
* f-strings
* Explicit types at boundaries
* Dataclasses for structured data

---

# 5. Error Handling

Forbidden patterns:

except: pass
except Exception: pass
except: continue
except Exception: continue

Every exception path must do one of the following:

* Raise a specific exception
* Return a typed error value
* Log with logger.exception and re-raise
* Handle a specific expected exception with explicit fallback logic

Broad exceptions require a justification comment:

BROAD_EXCEPT_OK: one sentence reason

---

# 6. Logging

No print statements in library or application code.

Use the standard logging module.

Each module must define:

```python
logger = logging.getLogger(__name__)
```

Logs must:

* State action and result
* Include identifiers
* Avoid vague language

Example:

```python
logger.info("Converted %s rows for dataset %s", n_rows, dataset_name)
logger.warning("Missing field %s for entity %s. Using default.", field, entity_id)
logger.exception("Failed to parse record %s", record_id)
```

---

# 7. ASCII Rule

Source code, documentation, comments, and logs must use ASCII only.

Forbidden:

* Greek symbols
* Mathematical glyphs
* Decorative Unicode
* Non-ASCII punctuation

Use plain ASCII words such as delta, tau, lambda.

Commits introducing non-ASCII characters must fail.

---

# 8. Placeholder Governance

No silent stubs.
No hidden TODO.
No placeholder returns such as `return None` unless correct behavior.

A placeholder must:

1. Raise:

   ```python
   raise NotImplementedError("PLACEHOLDER:P-001 short description")
   ```

2. Have an entry in `PLACEHOLDERS.md` including:

   * ID
   * File
   * Symbol
   * Owner
   * Reason
   * Exit condition

3. Have a failing test or explicit exception path referencing the same ID.

Untracked placeholders fail checks.

---

# 9. Documentation Standards

Documentation targets domain users first.

Documentation must:

* Explain workflows and decisions
* Include end-to-end examples
* Avoid autogenerated style

Documentation lives under:

docs/

Allowed markdown outside docs:

* README.md (root or package)
* CHANGELOG.md
* CONTRIBUTING.md
* LICENSE
* PLACEHOLDERS.md
* CONTRACT.md
* CLAUDE.md
* .cursor/ files

Minimum required docs:

docs/index.md
docs/overview.md
docs/quickstart.md
docs/concepts.md
docs/howto.md
docs/faq.md
docs/reference.md

Every new feature must update documentation.

---

# 10. Docstrings

All public modules, classes, and functions must include docstrings.

Docstrings must use Google style.

Docstrings must include:

* Purpose
* Arguments
* Returns
* Units when relevant
* Failure modes when relevant

Prefer strong naming over excessive comments.

---

# 11. Assistant Archive Rule

Non-trivial assistant outputs must be written to:

.cursor/ARCHIVE/

Filename format:

YYYYMMDD_HHMM_TOPIC_TAG.MD

Filenames must be ALL CAPS.

Each archive file must begin with:

* Title
* Goal
* Files touched
* Checks required

Non-trivial outputs include:

* Plans
* Contracts
* Architecture designs
* Runbooks
* Multi-step instructions
* Structural changes

Failure to archive fails checks.

---

# 12. Project Memory Rule

Durable decisions must live in:

docs/memory/

Important design decisions must not live only in chat.

Update memory after meaningful architectural changes.

---

# 13. Web Rules (If Applicable)

Tailwind only.
No CSS modules.
No styled-components.
No ad hoc CSS files.

Server components by default.
Client components only when required.
All user input must be validated.

---

# 14. Completion Standard

Work is complete only when:

* Local checks pass
* CI passes
* Tests pass
* No forbidden exception patterns exist
* No untracked placeholders exist
* No non-ASCII exists
* Documentation is updated
* Archive rule passes

No exceptions.
